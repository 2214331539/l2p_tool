# L2P-Tool Baseline for Continual Tool Retrieval

This repository contains only the source code needed to reproduce the current
L2P-Tool experiments. It does not include ToolHCL training/evaluation data,
Llama/SentenceTransformer model weights, checkpoints, replay buffers, or
metrics artifacts.

The implementation is an independent baseline path. It does not modify or
depend on ToolHCL hierarchy, box geometry, L1/L2 routing, or capability-space
retrieval. The model freezes a text encoder and trains:

- `prompt_keys`
- `prompt_pool`
- `fusion` MLP
- `classifier`

For each query, it selects top-k prompts by query-key cosine similarity,
mean-pools selected prompts, fuses prompt/query representations, and classifies
over all sampled or full visible tools.

## Source Scope

The reproducible source boundary is:

- `tool_l2p/data.py`
  - retrieval file discovery
  - ToolHCL-format sample parsing
  - mini-subset split creation/reuse
  - full-data tool mapping creation
  - dataset/collate utilities
- `tool_l2p/model.py`
  - frozen hashing encoder fallback
  - frozen SentenceTransformer encoder
  - frozen ToolHCL Llama embedding encoder
  - `ToolL2PModel`
  - global/local/sampled CE losses
  - checkpoint save/load
- `tool_l2p/train.py`
  - single-stage training
  - no-replay, regularized-no-replay, and replay variants
  - prompt balance/top-m sampling
  - classifier/fusion freeze controls
  - replay buffer build/load/save
  - current-task and old-task global evaluation after each stage
  - diagnostics and metrics writing
- `tool_l2p/eval.py`
  - checkpoint evaluation on global visible tools or task-local tools
- `tool_l2p/metrics.py`
  - Recall@K, NDCG@K, MRR
- `tool_l2p/summarize.py`
  - metrics summary for continual-learning runs
- `tool_l2p/compare.py`
  - no-replay / regularized-no-replay / replay comparison table
- `tool_l2p/utils.py`
  - stage ordering, JSON utilities, device resolution
- `train_tool_l2p.py`
  - wrapper that runs `base -> task1 -> task2 -> task3`
- `scripts/`
  - reproducible shell commands for the current experiments

Generated files such as `metrics.json`, replay buffers, checkpoints, and split
mappings are not committed. They are regenerated by the scripts.

## Expected Data Layout

Place ToolHCL data under `data/` with the same layout used by the original
project:

```text
data/train/raw/retrieval_train.json
data/train/raw/retrieval_eval.json
data/train/raw/train_tools_with_id.json
data/task1/raw/retrieval_train.json
data/task1/raw/retrieval_eval.json
data/task1/raw/task1_tools_with_id.json
data/task2/raw/retrieval_train.json
data/task2/raw/retrieval_eval.json

data/task3/raw/retrieval_train.json
```

The loader also tries ToolHCL-compatible fallback paths documented in
`tool_l2p/README.md`.

## Encoder Setup

For the current full-data experiments, the command uses:

```bash
--encoder_backend toolhcl_llm
--encoder_model auto
--llm_pooling embedding_only
```

`encoder_model=auto` resolves `ToolHCHL/config.py:LLAMA_PATH` when this code is
placed next to the original ToolHCL checkout. Otherwise it falls back to:

```text
ToolHCHL/models_hf/Meta-Llama-3-8B
```

You can override this path explicitly:

```bash
--encoder_model /path/to/Meta-Llama-3-8B
```

## Reproduce Current Replay Experiments

Set the Python executable and GPU:

```bash
export PY=/opt/miniconda3/envs/llama-factory/bin/python
export GPU=2
```

Run the 1-epoch replay baseline:

```bash
bash scripts/run_replay_ep1.sh
```

Run the 3-epoch replay baseline:

```bash
bash scripts/run_replay_ep3.sh
```

Run the 5-epoch replay baseline:

```bash
bash scripts/run_replay_ep5.sh
```

The main 5-epoch command uses the current setting:

```bash
CUDA_VISIBLE_DEVICES=$GPU $PY train_tool_l2p.py \
  --output_dir tool_l2p_runs_replay_ep5 \
  --full_data \
  --encoder_backend toolhcl_llm \
  --l2p_variant replay \
  --loss_type global_ce \
  --replay_strategy balanced_per_task \
  --replay_samples_per_task 2000 \
  --replay_ratio 0.25 \
  --batch_size 64 \
  --epochs 5 \
  --prompt_top_k 5 \
  --learning_rate 5e-4 \
  --save_replay_buffer
```

This keeps the replay upper bound fixed at 2000 samples per task.

## Summarize Results

```bash
$PY -m tool_l2p.summarize --metrics_path tool_l2p_runs_replay_ep5/metrics.json
```

Compare the 1/3/5 epoch replay runs:

```bash
$PY -m tool_l2p.compare \
  --no_replay_metrics tool_l2p_runs_replay/metrics.json \
  --regularized_metrics tool_l2p_runs_replay_ep3/metrics.json \
  --replay_metrics tool_l2p_runs_replay_ep5/metrics.json
```

## Notes on Current Observations

Increasing epochs improves current-task global evaluation substantially. With
the fixed replay cap used here, old-task global evaluation also improves but
remains limited, especially for the large base task. The reason is that
`balanced_per_task=2000` gives the base task only 2000 replay samples for over
10k tools, so most base tools are not replayed as positives in later stages.
